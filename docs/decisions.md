# Architecture Decisions

Running log of design decisions for this project. Every decision that affects how experiments run, how data is stored, or how results are reported goes here with reasoning.

---

## ADR-001: Two-script eval pipeline (build_index + run_eval)

Date: 2026-03
Status: Accepted

Decision: Split eval into two separate scripts.
- `src/eval/build_index.py` -- encodes all index profiles and saves to LanceDB. Run once per model/serialization combination.
- `src/eval/run_eval.py` -- loads existing LanceDB index, runs queries, computes metrics, writes results JSON.

Reason: Encoding 1M records takes 10-30 min depending on model. Separating index build from query eval means you build once and run eval as many times as needed (different top-k, different buckets, etc.) without re-encoding.

Rejected alternative: One end-to-end script. Simpler but wasteful -- every eval run re-encodes the full index.

## ADR-002: LanceDB for index storage

Date: 2026-03
Status: Accepted

Decision: Use LanceDB instead of raw FAISS index files for storing embeddings.

Reason:
- LanceDB stores vectors alongside metadata (entity_id, serialized text) in a single directory. No separate metadata file needed.
- Native Lance format is columnar (Parquet-compatible), so the index is also queryable via Polars/Pandas for debugging.
- Built-in ANN search (IVF_PQ, HNSW) without manually managing FAISS index parameters.
- Simpler to store and reload -- one directory per index, no custom serialization.
- Supports filtering (e.g. query top-K but filter by country) if needed later.

Index directory naming: `results/indexes/{model_key}_{serialization_format}_{quantization}/`

Examples:
- `results/indexes/gte_modernbert_base_pipe_fp32/`
- `results/indexes/bge_small_ft_pipe_binary/`
- `results/indexes/bm25_pipe/`  (special case -- inverted index, not vectors)

BM25 special case: BM25 does not use LanceDB. It uses rank_bm25 with a pickled index saved to `results/indexes/bm25_{serialization_format}/index.pkl`. Kept separate because BM25 is not vector-based.

## ADR-003: Results format -- JSON per run, CSV aggregate, Markdown report

Date: 2026-03
Status: Accepted

Decision: Three-layer results storage.

Layer 1 -- Per-run JSON (`results/{experiment_id}_{model}_{fmt}_{quant}.json`):
```json
{
  "experiment_id": "001",
  "model": "bm25",
  "serialization": "pipe",
  "mode": "zero_shot",
  "quantization": null,
  "dims": null,
  "index_size": 1000000,
  "timestamp": "2026-03-07T10:00:00Z",
  "overall": {
    "recall_at_1": 0.72,
    "recall_at_5": 0.89,
    "recall_at_10": 0.93,
    "mrr_at_10": 0.81,
    "ndcg_at_10": 0.83,
    "precision_at_5": 0.78
  },
  "per_bucket": {
    "pristine": {"recall_at_1": 0.95, "recall_at_5": 0.99, "recall_at_10": 0.99, "mrr_at_10": 0.97, "ndcg_at_10": 0.98, "precision_at_5": 0.94},
    "missing_firstname": {"recall_at_1": 0.71, ...},
    "missing_email_company": {"recall_at_1": 0.48, ...},
    "typo_name": {"recall_at_1": 0.31, ...},
    "domain_mismatch": {"recall_at_1": 0.62, ...},
    "swapped_attributes": {"recall_at_1": 0.55, ...}
  },
  "latency_ms": {
    "p50": 4.2,
    "p95": 8.1,
    "p99": 12.3,
    "n_queries": 1000
  },
  "index_size_mb": 384.2
}
```

Layer 2 -- Aggregate CSV (`results/master_results.csv`):
One row per experiment run. Columns: experiment_id, model, serialization, mode, quantization, dims, index_size, plus one column per metric per bucket (e.g. pristine_recall_at_1, typo_name_recall_at_10, etc.) plus latency columns.

Generated by `src/eval/aggregate.py` which reads all JSONs in results/ and builds the CSV.

Layer 3 -- Markdown report (`results/report.md`):
Human-readable summary generated by `src/eval/aggregate.py` alongside the CSV. Contains:
- Summary table: model vs bucket, recall@10, MRR@10
- Delta table: each model vs BM25 baseline (+ or - pp)
- Latency table
- Key findings section (auto-generated based on which model wins which bucket)

This is what gets shared on Monday.

## ADR-004: Model inference quirks documented per model

Date: 2026-03
Status: Accepted

Decision: Each model has different inference requirements. These are stored in configs/models.yaml (machine-readable) and UNDERSTANDING.md (human-readable). The encoder wrapper (src/models/encoder.py) reads from configs/models.yaml and applies them automatically. No hardcoded model-specific logic in eval scripts.

See UNDERSTANDING.md for full per-model details.

## ADR-005: Experiment logging convention

Date: 2026-03
Status: Accepted

Decision: Every experiment run is logged in three places:
1. `experiments/00N_name/config.json` -- what was run (model, data version, hyperparams)
2. `experiments/00N_name/notes.md` -- filled in after the run (observations, surprises, next steps)
3. `results/00N_*.json` -- machine-readable metrics

After each experiment:
- Fill notes.md with actual results and observations
- Update the experiment log table in README.md
- Commit with message: `exp(00N): <one-line result summary>`

Example commit: `exp(001): BM25 baseline -- R@10 pristine=0.94, typo_name=0.31`

Nothing gets committed without notes.md filled in.
